## 简介

SillyTavern Next 是一个基于现代 Web 技术栈构建的 AI 聊天应用，采用 GNU Affero General Public License v3.0 (AGPL-3.0) 许可证进行开源发布。本项目不仅提供了强大的功能特性，更重要的是通过 AGPL-3.0 许可证确保了软件的自由性和开源性，体现了开源社区的核心价值观。

- **强制开源**：任何基于该项目的衍生作品都必须同样开源
- **网络交互义务**：通过网络提供服务也视为分发，必须公开源码

- **版权声明**：要求保留版权声明和许可证文本

## GNU Affero General Public License v3.0 (AGPL-3.0) 详解

### AGPL-3.0 的核心条款

#### 1. 使用权利
- ✅ **自由使用**：任何人都可以自由使用、修改、复制和分发受许可的软件
- ✅ **商业使用**：允许将软件用于商业目的，无需支付许可费用
- ✅ **私人使用**：个人使用完全免费，无需授权

#### 2. 修改与分发义务
- ❗ **衍生作品开源**：基于本项目的任何修改或衍生作品都必须以 AGPL-3.0 或更高版本发布
- ❗ **源代码公开**：分发修改后的版本时，必须同时提供相应的源代码
- ❗ **许可证保持**：在分发时必须保留原始版权声明和许可证文本

#### 3. 网络服务特殊规定
- ⚠️ **网络交互视为分发**：通过网络提供服务（即使不下载客户端）也视为分发，必须公开源码
- ⚠️ **远程交互义务**：任何通过网络与用户交互的修改版本都必须提供源代码获取途径

### AGPL-3.0 与其他许可证的区别

#### 与 GPL-3.0 的区别
- **网络服务条款**：AGPL-3.0 特别针对网络服务器软件设计，要求通过网络提供服务时公开源码
- **适用范围**：主要适用于网络服务器软件，而 GPL-3.0 更广泛

#### 与 MIT/BSD 许可证的区别
- **copyleft 强度**：AGPL-3.0 是强 copyleft，MIT/BSD 是宽松许可证
- **衍生作品要求**：MIT/BSD 允许闭源衍生，AGPL-3.0 要求强制开源

### 实际影响分析

#### 对用户的影响
- **完全免费**：软件本身完全免费，无需购买许可证
- **功能完整**：可以自由使用所有功能，包括商业用途
- **修改自由**：可以根据需要修改软件以满足特定需求

#### 对企业的影响
- **开源义务**：如果企业基于本项目开发商业产品，必须开源其修改部分
- **网络服务合规**：如果企业通过网络提供基于本项目的 SaaS 服务，必须公开源码
- **专利风险**：获得专利授权保护，避免专利侵权风险

#### 对开发者的影响
- **贡献义务**：对项目做出的改进必须开源分享
- **协作机会**：促进开源社区协作和技术共享
- **知识产权保护**：通过 copyleft 保护项目不受专有化

## 社区贡献政策

### Issue 提交流程

#### 提交前准备工作
1. **搜索现有问题**：在提交新 Issue 前，务必搜索是否已有相同问题
2. **提供完整信息**：包括环境信息、操作步骤、期望行为与实际行为对比
3. **收集日志信息**：准备浏览器控制台日志和服务端日志

#### Issue 类型分类
- **Bug 报告**：包含重现步骤、预期结果、实际结果
- **功能请求**：详细描述需求背景和使用场景
- **性能问题**：提供性能测试数据和瓶颈分析
- **安全问题**：私密报告，避免公开披露

### Pull Request (PR) 提交流程

#### 开发工作流
```bash
# 1. Fork 并克隆仓库
git clone https://github.com/<your-username>/sillytavern-next.git
cd sillytavern-next

# 2. 创建功能分支
git checkout -b feat/your-feature

# 3. 开发与验证
npm install
npm run setup       # 初始化数据库
npm run dev
npm run typecheck   # 类型检查
npm run lint        # ESLint 检查

# 4. 提交更改
git commit -m "feat: 功能描述"

# 5. 推送到远程并发起 PR
git push origin feat/your-feature
```

#### 分支命名规范
| 前缀 | 用途 | 示例 |
|------|------|------|
| `feat/` | 新功能 | `feat/group-mention` |
| `fix/` | Bug 修复 | `fix/branch-creation-fail` |
| `refactor/` | 重构 | `refactor/extract-tags-editor` |
| `docs/` | 文档变更 | `docs/update-readme` |
| `chore/` | 构建/工具 | `chore/upgrade-deps` |
| `test/` | 测试相关 | `test/add-chat-store-test` |

#### Commit Message 规范
遵循 [Conventional Commits](https://www.conventionalcommits.org/) 标准：

```
<type>(<scope>): <subject>

<body>
```

**Type 类型**：
- `feat`：新功能
- `fix`：Bug 修复  
- `refactor`：重构
- `docs`：文档
- `style`：格式调整
- `perf`：性能优化
- `test`：测试
- `chore`：构建/工具

### 版本管理自动化

#### release-please 集成
项目集成了 [release-please](https://github.com/googleapis/release-please) 工具，实现版本号和 CHANGELOG 的自动化管理：

**版本号规则**（v0.x 阶段）：
- `fix:` `perf:` `refactor:` → patch：0.1.0 → 0.1.1
- `feat:` → minor：0.1.0 → 0.2.0
- `feat!:` `fix!:` 或包含 `BREAKING CHANGE:` → minor：0.1.0 → 0.2.0

**工作流程**：
1. PR 合并到 `main` 后，自动创建或更新 `chore(main): release x.y.z` PR
2. 自动包含版本提升、CHANGELOG 更新、`.release-please-manifest.json` 同步
3. Maintainer 审核后合并，自动打标签并创建 GitHub Release

### 代码风格规范

#### 组件组织原则
- `src/components/ui/` — 通用基础 UI 组件（按钮、输入框等）
- `src/components/<feature>/` — 按功能模块分组的组件
- 仅在两个或多个页面引用时才提取为共享组件
- 大组件（>200 行）拆分为独立文件

#### TypeScript 规范
- 优先使用 `interface` 描述对象类型，`type` 用于联合/工具类型
- 共享类型放在 `src/types/index.ts`
- 避免使用 `any` 和 `as unknown as`，必要时扩展类型定义
- 异步函数必须正确处理 `await` 或显式 `void` / `.catch()` 处理

#### 错误处理规范
- 关键操作（保存、同步）：`try/catch` + 用户可见的错误提示
- 非关键操作（加载建议、装饰性请求）：`.catch(console.warn)` 至少有日志
- 禁止 `.catch(() => {})` 完全吞掉异常

#### Magic Number 规范
定时器、阈值、限制值等数字常量应集中在 `src/lib/constants/` 目录下。

## 常见问题解答

### AGPL-3.0 与 MIT 许可证有什么区别？

**AGPL-3.0（强 copyleft）**：
- 要求衍生作品必须开源
- 网络服务也视为分发，必须公开源码
- 保护用户获得源码的权利

**MIT 许可证（宽松）**：
- 允许闭源衍生
- 网络服务不需要公开源码
- 最小限制，最大自由

### 我可以在商业产品中使用 AGPL-3.0 代码吗？

**可以，但有条件**：
- 如果仅在本地使用，无需开源
- 如果通过网络提供服务，必须公开源码
- 如果分发修改版本，必须开源
- 必须保留版权声明和许可证

### 如何确保合规？

1. **仔细阅读许可证全文**
2. **咨询法律专业人士**
3. **建立内部合规流程**
4. **定期审查使用情况**
5. **准备开源贡献**

### 什么是 copyleft？

Copyleft 是一种版权策略，通过许可证条款确保软件的自由使用、修改和分发权利：
- **自由软件**：用户拥有使用、研究、修改和分发的自由
- **源码公开**：修改后的版本必须开源
- **禁止专有化**：防止软件被私有化
- **社区共享**：促进技术共享和协作

### 网络服务中的 AGPL-3.0

AGPL-3.0 的独特之处在于它要求通过网络提供服务也必须公开源码：
- **远程交互**：用户通过网络与程序交互
- **源码提供**：必须提供修改版本的源码
- **用户权利**：确保用户获得源码使用权
- **商业影响**：对 SaaS 模式有重大影响
